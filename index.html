Lektionen

Jedes Modul kombiniert Bild, Regel, Beispiel und Challenge.

So entsteht Verstaendnis statt Auswendiglernen.

Modul 1

Syntax und Datentypen

Lektion 1: Atome, Zahlen, Strings

Erlang unterscheidet klar zwischen Atomen wie ok, Zahlen wie 42 und Binaries wie <<"hi">>.

Merksatz: Ein Atom beschreibt oft Bedeutung, ein Binary oft echte Daten.

State = idle,
Count = 3,
Name = <<"Ada">>.

Lektion 2: Pattern Matching

Pattern Matching ist kein "Vergleichen", sondern strukturiertes Entpacken von Werten.

Merksatz: Erlang fragt immer: "Passt die Form?"

{ok, User} = {ok, #{name => <<"Ada">>}}.
Mini-Challenge

Welche Teile koennen in {error, timeout} direkt mit Pattern Matching extrahiert werden?

Modul 2

Funktionen, Guards und Rekursion

Lektion 3: Mehrere Funktionskoepfe

Eine Funktion kann mehrere Formen haben. Erlang prueft sie von oben nach unten.

Merksatz: Erst Pattern, dann Guard, dann Funktionskoerper.

size([]) -> 0;
size([_ | Tail]) -> 1 + size(Tail).

Lektion 4: Guards

Guards machen Regeln lesbar und verhindern versteckte Logik im Body.

Merksatz: Guards klaeren Faelle, bevor Arbeit beginnt.

kind(N) when N > 0 -> positive;
kind(0) -> zero;
kind(_) -> negative.

Lektion 5: Rekursion statt Schleife

In Erlang ist Rekursion die normale Art, ueber Daten zu laufen.

Merksatz: Jede Rekursion braucht Start, Schritt und Ende.

sum([]) -> 0;
sum([H | T]) -> H + sum(T).
Mini-Challenge

Schreibe gedanklich eine Funktion all_even/1, die nur dann true liefert, wenn alle Elemente gerade sind.

Modul 3

Prozesse und Message Passing

Lektion 6: Prozesse starten

Ein Erlang-Prozess ist extrem leichtgewichtig und hat seinen eigenen Zustand.

Merksatz: Teile Zustand nicht, schicke Nachrichten.

Pid = spawn(fun loop/0).

Lektion 7: Nachrichten senden

Mit ! sendest du Messages asynchron in die Mailbox eines Prozesses.

Merksatz: Senden ist sofort, Verarbeiten spaeter.

Pid ! {self(), ping}.

Lektion 8: Receive-Loops

Ein Prozess wird meist als Schleife modelliert: Nachricht empfangen, reagieren, weitermachen.

Merksatz: Verhalten entsteht aus Empfang + naechstem Loop.

receive
    stop -> ok;
    Msg -> io:format("~p~n", [Msg]), loop()
end.
Mini-Challenge

Wie wuerdest du einen Counter-Prozess entwerfen, der auf increment und get reagiert?

Modul 4

OTP und Fehlertoleranz

Lektion 9: Let it crash

In Erlang ist Fehlerbehandlung oft strukturell organisiert: Prozesse duerfen crashen, andere Prozesse stellen sie wieder her.

Merksatz: Nicht jeder Fehler wird im selben Prozess gefixt.

link(WorkerPid).

Lektion 10: GenServer-Denken

GenServer kapselt Zustand und standardisiert das Nachrichtenschema.

Merksatz: OTP gibt dir Struktur fuer wiederkehrende Prozessmuster.

handle_call(get, _From, State) ->
    {reply, State, State}.

Lektion 11: Supervisors

Supervisors entscheiden, ob einzelne Worker oder ganze Gruppen neu gestartet werden.

Merksatz: Robustheit ist eine Baumstruktur, kein Zufall.

{one_for_one, 5, 10}
Mini-Challenge

Wann waere eine one_for_one-Strategie sinnvoller als one_for_all?

Modul 5

Concurrency Patterns

Lektion 12: Worker Pools

Mehrere Prozesse koennen dieselbe Aufgabe parallel bearbeiten, ohne sich gegenseitig zu blockieren.

Merksatz: Parallelitaet braucht Aufgabenteilung, nicht nur mehr CPU.

Workers = [spawn(fun worker/0) || _ <- lists:seq(1, 4)].

Lektion 13: State isolieren

Zustand lebt idealerweise nur in einem Prozess und wird nie direkt geteilt.

Merksatz: Isolation reduziert Nebenwirkungen.

store_loop(State) ->
    receive
        {put, Key, Val} -> store_loop(maps:put(Key, Val, State));
        {get, From, Key} -> From ! maps:get(Key, State), store_loop(State)
    end.
Mini-Challenge

Skizziere eine Pipeline mit drei Prozessen: empfangen, pruefen, speichern.

Modul 6

Distributed Erlang

Lektion 14: Nodes

Ein Node ist eine laufende Erlang-Instanz, die mit anderen Nodes kooperieren kann.

Merksatz: Verteilung beginnt mit expliziter Verbindung, nicht mit Magie.

net_kernel:connect_node('worker@host').

Lektion 15: Remote Messaging

Nachrichten koennen ueber Nodes hinweg verschickt werden, wenn Namensgebung und Verbindung stimmen.

Merksatz: Ein verteiltes System ist nur so gut wie seine Fehlerannahmen.

{service, 'worker@host'} ! ping.

Lektion 16: Praxisdenken

Bei Verteilung zaehlen Timeouts, Retries, Ueberwachung und klare Verantwortlichkeiten.

Merksatz: Erfolgreiche Systeme planen Netzwerkfehler fest ein.

receive
    reply -> ok
after 2000 ->
    timeout
end.
Mini-Challenge

Welche Fehler koennen auftreten, wenn ein Remote-Node gerade nicht erreichbar ist?

FAQ

Die wichtigsten Fragen fuer konstantes Vorankommen.

Wie lerne ich mit diesem Kurs am effektivsten?

Lies nie nur passiv. Arbeite immer in der Reihenfolge: Lektion, Merksatz wiederholen, Challenge denken, Quiz beantworten.

Was mache ich, wenn sich Rekursion ungewohnt anfuehlt?

Male Eingabe, naechsten Schritt und Endfall auf Papier. Rekursion wird klar, wenn du den Datenfluss siehst.

Wann bin ich bereit fuer OTP?

Wenn du Prozesse, Nachrichten und zustandsbehaftete Loops mental sicher modellieren kannst. Dann macht OTP ploetzlich Sinn.

Ist der Kurs eher fuer Einsteiger oder fortgeschrittene Entwickler?

Beides. Die Progression ist anfängerfreundlich, aber die spaeteren Module fuehren bereits an produktionsnahe Konzepte heran.
